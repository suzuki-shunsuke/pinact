# Ref does not resolve to a tag

By default, pinact only pins refs that resolve to tags. If a ref does not
resolve to a tag — for example, the repository has a branch of that name but no
corresponding tag — pinact raises an error rather than silently pinning to the
mutable branch tip.

Branches are mutable by design; tags are mutable by exception. Pinning a
branch captures a snapshot of whatever HEAD happened to be at resolution time,
while the upstream author may push new commits or force-push the branch at any
moment. Pinact's output suggests a fixed "version," but a branch-sourced pin
carries no such promise. Silently pinning a branch masks this distinction and
can lead to pins drifting unexpectedly on re-resolution.

## Example

```console
$ pinact run
ERRO[0000] parse a line                                  action=allenheltondev/detect-breaking-changes-action error="get a reference: ref does not resolve to a tag and branch pinning is disabled" help_docs="https://github.com/suzuki-shunsuke/pinact/blob/main/docs/codes/005.md" owner=allenheltondev ref=v1 repo=detect-breaking-changes-action workflow_file=.github/workflows/example.yaml
```

## Resolution

In preference order:

1. **Switch to an action that publishes tags.** Lowest supply-chain risk and
   matches pinact's default contract.
2. **Enable branch pinning explicitly.** If the action is legitimately
   branch-only and you accept the mutability trade-off, set one of the
   following. Pinact will then resolve the ref via the bare (non-tag)
   namespace, log a warning, and pin to the branch HEAD.

   - CLI flag: `pinact run --allow-branch-pins`
   - Config (`.pinact.yaml`):
     ```yaml
     version: 3
     allow_branch_pins: true
     ```
   - `pinact-action` input: `allow_branch_pins: "true"`
3. **Pin manually** to a specific commit SHA if neither of the above suits.
